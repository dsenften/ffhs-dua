== Übersicht

Diese Dokumentation beschreibt die Testfälle für die Stack-Implementierungen im `algs4.fundamentals.stack`-Modul. Es werden drei verschiedene Stack-Implementierungen getestet:

1. `Stack` - Eine Implementierung mit verketteter Liste
2. `FixedCapacityStack` - Eine Implementierung mit fester Kapazität
3. `ResizingArrayStack` - Eine Implementierung mit dynamisch anpassbarer Array-Größe

== Testfälle für Stack-Klasse

Die Testfälle decken folgende Aspekte der Stack-Implementierung ab:

=== Grundfunktionen

==== Leerer Stack (`test_empty_stack`)

Überprüft, ob ein neu erstellter Stack leer ist:

* `is_empty()` gibt `True` zurück
* `size()` gibt `0` zurück
* `len()` gibt `0` zurück

==== Push und Pop (`test_push_pop`)

Testet die grundlegenden Operationen zum Hinzufügen und Entfernen von Elementen:

* Elemente werden korrekt hinzugefügt
* Die Grösse des Stacks wird korrekt aktualisiert
* Elemente werden in der LIFO-Reihenfolge (Last-In-First-Out) entfernt
* Nach dem Entfernen aller Elemente ist der Stack leer

=== Hilfsfunktionen

==== Peek-Funktion (`test_peek`)

Überprüft die Funktionalität der `peek()`-Methode:

* Gibt das oberste Element zurück, ohne es zu entfernen
* Nach dem Entfernen des obersten Elements gibt `peek()` das nächste Element zurück

==== String-Repräsentation (`test_repr`)

Testet die String-Repräsentation des Stacks:

* Die Elemente werden in der richtigen Reihenfolge (LIFO) dargestellt
* Das Format entspricht den Erwartungen

=== Iterator-Funktionalität

==== Iterator-Test (`test_iterator`)

Überprüft die Iterator-Funktionalität des Stacks:

* Die Iteration erfolgt in der richtigen Reihenfolge (LIFO)
* Der Stack bleibt nach der Iteration unverändert

=== Ausnahmefälle und Fehlerbehandlung

==== Pop auf leerem Stack (`test_exception_pop_empty`)

Testet, ob eine `ValueError` geworfen wird, wenn `pop()` auf einem leeren Stack aufgerufen wird.

==== Peek auf leerem Stack (`test_exception_peek_empty`)

Testet, ob eine `ValueError` geworfen wird, wenn `peek()` auf einem leeren Stack aufgerufen wird.

=== Verschiedene Datentypen

==== Verschiedene Datentypen (`test_different_types`)

Überprüft, ob der Stack mit verschiedenen Datentypen funktioniert:

* Integer
* String
* Listen

== Testfälle für FixedCapacityStack-Klasse

=== Grundfunktionen

==== Leerer Stack (`test_empty_stack`)

Überprüft, ob ein neu erstellter FixedCapacityStack leer ist.

==== Push und Pop (`test_push_pop`)

Testet die grundlegenden Push- und Pop-Operationen für den FixedCapacityStack.

=== Kapazität

==== Kapazitätstest (`test_capacity`)

Überprüft, ob der FixedCapacityStack die angegebene Kapazität einhält:

* Der Stack kann bis zur angegebenen Kapazität gefüllt werden
* Die Elemente werden in der richtigen Reihenfolge entfernt

== Testfälle für ResizingArrayStack-Klasse

=== Grundfunktionen

==== Leerer Stack (`test_empty_stack`)

Überprüft, ob ein neu erstellter ResizingArrayStack leer ist.

==== Push und Pop (`test_push_pop`)

Testet die grundlegenden Push- und Pop-Operationen für den ResizingArrayStack.

=== Größenanpassung

==== Resize-Test (`test_resize`)

Überprüft die automatische Größenanpassung des ResizingArrayStacks:

* Die interne Array-Größe wird erhöht, wenn mehr Elemente hinzugefügt werden
* Die interne Array-Größe wird verkleinert, wenn Elemente entfernt werden

=== Iterator-Funktionalität

==== Iterator-Test (`test_iterator`)

Überprüft die Iterator-Funktionalität des ResizingArrayStacks:

* Die Iteration erfolgt in der richtigen Reihenfolge (LIFO)
* Der Stack bleibt nach der Iteration unverändert

== Testausführung

Die Tests können mit dem Python-Unittest-Framework ausgeführt werden:

[source,bash]
----
# Ausführung der Tests
python -m unittest tests.test_stack

# Alternativ mit pytest
python -m pytest tests/test_stack.py -v
----

== Testabdeckung

Die Tests decken folgende Aspekte der Stack-Implementierungen ab:

* Grundlegende Operationen (push, pop)
* Hilfsfunktionen (is_empty, size, peek)
* Iterator-Funktionalität
* String-Repräsentation
* Ausnahmefälle und Fehlerbehandlung
* Verschiedene Datentypen
* Kapazitätsverwaltung (für FixedCapacityStack)
* Dynamische Größenanpassung (für ResizingArrayStack)

== Best Practices

Die Tests folgen diesen Best Practices für Python 3:

* Verwendung des `unittest`-Frameworks
* Klare Testfallnamen, die den Zweck des Tests beschreiben
* Aussagekräftige Docstrings für jeden Test
* Verwendung von `setUp` für die Initialisierung vor jedem Test
* Überprüfung von Ausnahmen mit `assertRaises`
* Tests für verschiedene Datentypen mit generischen Typen
* Vollständige Abdeckung aller öffentlichen Methoden
